1.1 Latar Belakang
Perkembangan teknologi informasi dan komunikasi saat ini telah memberikan dampak yang signifikan dalam berbagai bidang, termasuk dalam bidang pelayanan kesehatan. Namun dalam praktiknya, masih banyak warga yang mengalami kesulitan dalam menyampaikan pengaduan, mengakses layanan administrasi, maupun memantau status permohonan mereka. Selain itu, pengelolaan data pengaduan dan layanan administrasi masih belum terorganisir dengan baik dalam suatu sistem terpusat.
Berdasarkan permasalahan tersebut, diperlukan suatu sistem yang dapat membantu mengelola proses pelayanan masyarakat secara efektif dan efisien. BAB II 
LANDASAN TEORI
2.1. Object-Oriented Programming (OOP)
Object-Oriented Programming (OOP) merupakan paradigma pemrograman yang menggunakan objek sebagai dasar dalam pengembangan sistem. Pendekatan ini bertujuan untuk membuat program lebih terstruktur, mudah dikembangkan, serta mudah dipelihara.
2.2. Encapsulation
Encapsulation merupakan konsep dalam Object-Oriented Programming (OOP) yang digunakan untuk membungkus data (atribut) dan method dalam satu kesatuan yang disebut class. Konsep ini juga berfungsi untuk menyembunyikan detail internal suatu objek dari akses langsung pihak luar (information hiding), sehingga data hanya dapat diakses melalui method tertentu yang telah disediakan. Dengan adanya encapsulation, keamanan data menjadi lebih terjaga dan perubahan data dapat dikontrol melalui proses tertentu.
 Contoh Penerapan:
Dalam Sistem Pelayanan Masyarakat, terdapat class Donor yang memiliki atribut seperti nama, golonganDarah, nomorTelepon, dan alamat. Atribut tersebut bersifat private, sehingga tidak dapat diakses langsung dari luar class. Untuk mengakses dan mengubah data tersebut, digunakan method seperti getNama(), setNama(), getGolonganDarah(), dan setNomorTelepon(). Dengan demikian, data donor tetap aman dan hanya dapat dimodifikasi melalui method yang telah ditentukan.
2.3. Inheritance
Inheritance adalah konsep dalam Object-Oriented Programming (OOP) yang memungkinkan suatu class mewarisi atribut dan method dari class lain. Dalam inheritance terdapat dua jenis class, yaitu superclass sebagai kelas induk dan subclass sebagai kelas turunan yang mewarisi atribut serta method dari superclass. Konsep ini digunakan untuk mengurangi duplikasi kode serta meningkatkan efisiensi dalam pengembangan sistem.
Contoh Penerapan:
Dalam sistem ini terdapat class User sebagai superclass yang memiliki atribut seperti id, nama, email, dan password. Kemudian dibuat subclass Admin dan Masyarakat yang mewarisi atribut dari class User. Class Admin memiliki fungsi tambahan seperti mengelola data lokasi, data layanan, dan permohonan layanan, sedangkan class Masyarakat digunakan untuk melakukan request darah dan melihat riwayat donor. Dengan inheritance, tidak perlu menuliskan ulang atribut yang sama di setiap class.
2.4. Polymorphism
Polymorphism adalah kemampuan suatu method untuk memiliki lebih dari satu bentuk atau implementasi dalam Object-Oriented Programming (OOP). Dengan polymorphism, satu method dapat digunakan dalam berbagai kondisi dengan perilaku yang berbeda.
Polymorphism terdiri dari overloading dan overriding.
Overloading
Overloading adalah penggunaan method dengan nama yang sama tetapi memiliki parameter yang berbeda dalam satu class.
Contoh Penerapan:
Pada class DonationService, terdapat method:
tambahDonation(donorId, lokasiId)
tambahDonation(donorId, lokasiId, jumlahDarah)
Kedua method tersebut memiliki nama yang sama tetapi parameter berbeda, sehingga memudahkan fleksibilitas dalam penggunaan.
Overriding
Overriding adalah proses mendefinisikan kembali method yang sudah ada pada superclass di dalam subclass dengan implementasi yang berbeda.
Contoh Penerapan:
Class User memiliki method login(), kemudian di subclass Admin method tersebut dioverride untuk menambahkan validasi khusus admin, sedangkan pada subclass Masyarakat method login() digunakan untuk akses pengguna biasa.
2.5. Abstraction
Abstraction adalah proses penyederhanaan objek dengan hanya menampilkan bagian penting dan menyembunyikan detail implementasinya. Konsep ini digunakan untuk mengurangi kompleksitas sistem sehingga memudahkan dalam pengembangan dan pemeliharaan aplikasi.
 Contoh Penerapan:
Dalam sistem ini digunakan interface DonationService yang hanya mendefinisikan method seperti:
tambahDonation()
getRiwayatDonor()
Sedangkan implementasi detailnya terdapat pada class DonationServiceImpl. Dengan abstraction, pengguna sistem hanya mengetahui fungsi utama tanpa perlu memahami proses internal bagaimana data diproses atau disimpan.
2.5.1 Interface
Interface merupakan kumpulan deklarasi method yang digunakan sebagai kontrak bagi class lain. Interface hanya berisi definisi method tanpa implementasi sehingga class yang mengimplementasikannya wajib menyediakan implementasi dari method tersebut. Penggunaan interface membantu meningkatkan fleksibilitas, modularitas, dan kemudahan pengembangan sistem.
Contoh Penerapan:
Dalam Sistem Manajemen Donor Darah digunakan interface DonationService yang mendefinisikan beberapa method seperti:
â-? tambahDonation()
â-? getDonationById()
â-? getRiwayatDonor()
Implementasi dari method-method tersebut dilakukan pada class DonationServiceImpl. Dengan menggunakan interface, perubahan implementasi dapat dilakukan tanpa mempengaruhi class lain yang menggunakan layanan tersebut.
2.5.2 Abstract Class
Abstract Class merupakan class yang tidak dapat dibuat objeknya secara langsung dan digunakan sebagai dasar bagi class turunan. Abstract class dapat memiliki atribut, method biasa, maupun abstract method yang harus diimplementasikan oleh subclass.
Contoh Penerapan:
Dalam sistem ini digunakan class User sebagai abstract class yang memiliki atribut umum seperti id, nama, email, dan password. Class Admin dan Masyarakat merupakan turunan dari class User yang mewarisi atribut dan method tersebut serta menambahkan fungsi sesuai kebutuhan masing-masing.
2.5.3 Exception Handling
Exception Handling merupakan mekanisme penanganan kesalahan yang terjadi saat program dijalankan agar aplikasi tetap berjalan dengan baik dan tidak berhenti secara tiba-tiba. Penanganan exception dilakukan menggunakan blok try-catch atau custom exception.
Contoh Penerapan:
Dalam Sistem Manajemen Donor Darah, exception handling digunakan ketika pengguna mencari data donor atau data lokasi yang tidak ditemukan pada database. Sistem akan menampilkan pesan kesalahan yang sesuai melalui custom exception seperti ResourceNotFoundException sehingga pengguna mendapatkan informasi yang jelas mengenai kesalahan yang terjadi.
2.6. Bahasa Pemograman Java
Java merupakan bahasa pemrograman berbasis Object-Oriented Programming (OOP) yang digunakan dalam pengembangan aplikasi. Java memiliki keunggulan seperti portabilitas (dapat dijalankan di berbagai platform), keamanan yang baik, serta dukungan library yang luas sehingga banyak digunakan dalam pengembangan sistem berbasis web.
 Contoh Penerapan:
Pada Sistem Pelayanan Masyarakat, Java digunakan untuk membuat class seperti User, Donor, Donation, Location, dan BloodRequest. Konsep OOP seperti encapsulation, inheritance, dan polymorphism diterapkan dalam pengelolaan data dan logika sistem.
2.7. Spring Boot
Spring Boot adalah framework berbasis Java yang digunakan untuk membangun aplikasi web dengan konfigurasi otomatis. Framework ini mempermudah proses pengembangan karena menyediakan fitur seperti dependency management dan auto-configuration, sehingga pengembang tidak perlu melakukan konfigurasi secara manual.
Contoh Penerapan:
Dalam sistem ini, Spring Boot digunakan untuk membangun backend aplikasi berupa REST API. Class seperti DonorController, DonationController, dan BloodRequestController digunakan untuk menangani request dari client, sedangkan Service digunakan untuk mengelola logika bisnis.
2.8 Spring Data JPA
Spring Data JPA merupakan bagian dari Spring Framework yang digunakan untuk mengelola database menggunakan pendekatan Object Relational Mapping (ORM). Dengan JPA, pengembang dapat berinteraksi dengan database melalui objek tanpa harus menulis query SQL secara manual.
 Contoh Penerapan:
Pada sistem ini, setiap tabel seperti donor, donation, dan location direpresentasikan sebagai class Entity. Repository seperti DonorRepository dan DonationRepository digunakan untuk melakukan operasi database seperti menyimpan, mengambil, dan menghapus data tanpa menulis query SQL secara langsung.

2.10 Database (MySQL) 
Database merupakan kumpulan data yang disimpan secara terstruktur dan digunakan untuk mendukung sistem. Pada sistem ini digunakan MySQL sebagai database karena memiliki performa yang baik, stabil, serta mudah digunakan dalam pengembangan aplikasi berbasis web.

BAB III 
PERANCANGAN SISTEM
3.1 Studi Kasus
Sistem yang dirancang adalah Sistem Pelayanan Masyarakat (SmartPelayanan) . Sistem ini bertujuan untuk membantu masyarakat dalam memperoleh informasi dan layanan terkait pelayanan masyarakat secara lebih cepat, transparan, dan efisien.
Saat ini, proses pelayanan administrasi dan penanganan pengaduan masyarakat di banyak instansi masih dilakukan secara manual atau belum terintegrasi dengan baik. Masyarakat sering mengalami kesulitan dalam menyampaikan keluhan, mengetahui status permohonan, serta mengakses layanan administrasi secara cepat. Selain itu, belum tersedia sistem yang dapat mengelola pengaduan dan layanan masyarakat secara terpadu dalam satu platform.
3.2 Fitur Aplikasi
Sistem Pelayanan Masyarakat Berbasis Web (SmartPelayanan) dirancang untuk meningkatkan kualitas pelayanan masyarakat melalui digitalisasi layanan pelayanan masyarakat. Sistem ini menyediakan berbagai fitur yang terintegrasi untuk mendukung proses pengelolaan data, pelayanan permohonan layanan, serta penyampaian informasi secara efektif.
Berikut merupakan fitur-fitur utama yang terdapat dalam sistem:
3.2.1 Fitur Autentikasi Pengguna (Login & Registrasi)
Fitur autentikasi digunakan untuk mengelola akses pengguna ke dalam sistem. Setiap pengguna diwajibkan melakukan registrasi akun sebelum dapat menggunakan layanan yang tersedia.
Sistem membedakan hak akses berdasarkan peran pengguna, yaitu Admin dan Masyarakat. Proses login dilakukan menggunakan email dan password yang telah terdaftar. Setelah berhasil login, pengguna akan diarahkan ke halaman sesuai dengan perannya masing-masing.
3.2.2 Fitur Manajemen Data Donor
Fitur ini digunakan untuk mengelola data individu yang bersedia menjadi pelayanan masyarakat. Data donor disimpan secara terstruktur sehingga dapat digunakan sebagai sumber informasi dalam proses pelayanan.
Admin memiliki hak penuh untuk menambah, mengubah, menghapus, dan melihat data donor. Data yang dikelola meliputi identitas lengkap seperti nama, nama orang tua, nomor telepon, jenis kelamin, email, kategori layanan, dan alamat.
3.2.3 Fitur Informasi Lokasi Donor Darah (instansi pelayanan/Rumah Sakit)
Fitur ini menyediakan informasi mengenai lokasi tempat pelayanan masyarakat yang dapat diakses oleh masyarakat.
Sistem menampilkan daftar lokasi pelayanan masyarakat yang mencakup nama tempat, alamat, serta wilayah. Selain itu, setiap lokasi juga dilengkapi dengan koordinat geografis berupa latitude dan longitude untuk mendukung integrasi dengan peta digital.
3.2.4 Fitur Integrasi Peta (Google Maps)
Fitur ini merupakan pengembangan dari informasi lokasi pelayanan masyarakat dengan menampilkan lokasi secara visual melalui peta digital.
Pengguna dapat melihat posisi lokasi pelayanan masyarakat dalam bentuk marker pada peta. Selain itu, sistem juga menyediakan fitur navigasi yang memungkinkan pengguna mendapatkan petunjuk arah menuju lokasi pelayanan masyarakat melalui Google Maps. Dengan adanya fitur ini, masyarakat dapat lebih mudah menemukan lokasi pelayanan masyarakat secara akurat.
3.2.5  Fitur Manajemen Stok Darah
Fitur ini digunakan untuk mengelola data ketersediaan data layanan pada setiap lokasi donor. Admin dapat memperbarui jumlah data layanan berdasarkan kategori layanan tertentu seperti A, B, AB, dan O. Informasi stok ini dapat diakses oleh masyarakat sehingga dapat membantu dalam pengambilan keputusan saat melakukan permohonan layanan.
3.2.6  Fitur Layanan Permintaan Darah
Fitur ini memungkinkan masyarakat untuk mengajukan permohonan layanan secara online.
Pengguna dapat mengisi data permintaan seperti kategori layanan yang dibutuhkan, jumlah darah, serta lokasi tujuan pengambilan. Permintaan yang diajukan akan diproses oleh Admin untuk selanjutnya ditindaklanjuti.
3.2.7 Fitur Status Pelayanan
Fitur ini digunakan untuk memberikan informasi terkait perkembangan proses permohonan layanan.
Setiap permintaan akan memiliki status tertentu, yaitu:
Pending (permintaan baru)
Diproses (sedang ditangani)
Selesai (permintaan telah terpenuhi)
Pengguna dapat memantau status tersebut secara real-time melalui sistem.
3.2.8 Fitur Riwayat Pelayanan
Fitur ini memungkinkan masyarakat untuk menyampaikan keluhan, kritik, maupun saran terhadap pelayanan donor darah.Pengaduan yang masuk akan dikelola oleh Admin, yang dapat memberikan tanggapan atau tindakan lanjutan. Fitur ini menjadi sarana komunikasi antara masyarakat dan pengelola sistem.
3.2.9 Fitur Pengaduan Masyarakat
Fitur ini memungkinkan masyarakat untuk menyampaikan keluhan, kritik, maupun saran terhadap pelayanan masyarakat.
Pengaduan yang masuk akan dikelola oleh Admin, yang dapat memberikan tanggapan atau tindakan lanjutan. Fitur ini menjadi sarana komunikasi antara masyarakat dan pengelola sistem.
3.2.10 Fitur Riwayat  Pendonor
Fitur ini digunakan untuk mencatat aktivitas pelayanan masyarakat yang telah dilakukan oleh pendonor. Data yang dicatat meliputi tanggal donor, lokasi donor, jumlah darah, dan status donor. Fitur ini memungkinkan pengguna melihat riwayat donor yang pernah dilakukan.

Database digunakan untuk menyimpan seluruh data yang dibutuhkan seperti data pengguna, donor, lokasi instansi pelayanan, permohonan layanan, data layanan, pengaduan, serta riwayat pelayanan.

3.4 Desain OOP & Rancangan API 
Desain OOP (Object-Oriented Programming) adalah pendekatan dalam perancangan sistem yang memodelkan aplikasi sebagai kumpulan objek (class) yang saling berinteraksi. Setiap objek merepresentasikan entitas nyata dalam sistem, seperti pengguna, donor, atau permohonan layanan.
3.4.1 Class Diagram
Class diagram merupakan representasi visual dari struktur OOP yang menggambarkan class, atribut, method, serta hubungan antar class seperti inheritance, association, dan composition. Diagram ini membantu pengembang memahami struktur sistem sebelum diimplementasikan ke dalam kode program.
Gambar 3.4.1 Class Diagram Donor Darah
3.4.2 Hubungan antar class
Inheritance (Admin terhadap User)
Class Admin memiliki hubungan inheritance dengan class User, yang berarti Admin merupakan turunan dari User dan mewarisi seluruh atribut umum seperti id, nama, email, dan password. User dibuat sebagai abstract class karena hanya berfungsi sebagai dasar bagi class lain, sedangkan Admin menambahkan fungsi khusus seperti mengelola data dalam sistem.
Association antara User dan Donor Class User memiliki hubungan association dengan class Donor dengan kardinalitas satu banding satu, yang berarti setiap user memiliki satu data donor. Donor menyimpan informasi tambahan seperti kategori layanan, alamat, dan tanggal lahir yang tidak terdapat pada User.
Association antara Donor dan Donation Class Donor berhubungan dengan class Donation dalam relasi satu ke banyak, yang berarti satu donor dapat melakukan beberapa kali kegiatan pelayanan masyarakat, sedangkan setiap data Donation hanya dimiliki oleh satu Donor.
Association antara Donation dan Location Class Donation memiliki hubungan dengan class Location dalam bentuk banyak ke satu, yang menunjukkan bahwa beberapa kegiatan donor dapat dilakukan di satu lokasi yang sama, seperti rumah sakit atau instansi pelayanan.
Association antara User dan BloodRequest Class User memiliki hubungan satu ke banyak dengan class BloodRequest, karena satu user dapat membuat beberapa permohonan layanan sesuai kebutuhan.
Composition antara BloodRequest dan ServiceHistory Class BloodRequest memiliki hubungan composition dengan ServiceHistory, yang berarti ServiceHistory merupakan bagian dari BloodRequest dan bergantung sepenuhnya pada keberadaannya. Jika BloodRequest dihapus, maka seluruh data ServiceHistory yang terkait juga akan ikut terhapus.
3.4.3 Identifikasi Layer Sistem
@Entity (Representasi database) Class User, Admin, Donor, Donation, BloodRequest, ServiceHistory, dan Location termasuk dalam kategori @Entity karena merepresentasikan tabel dalam database dan digunakan untuk menyimpan data sistem.
@Repository (Akses data) Class seperti UserRepository, DonorRepository, DonationRepository, BloodRequestRepository, ServiceHistoryRepository, dan LocationRepository termasuk dalam @Repository karena berfungsi untuk mengakses database dan menjalankan query seperti pencarian data menggunakan metode findBy.
@Service (Logika bisnis) Class seperti UserService, DonorService, DonationService, BloodRequestService, ServiceHistoryService, dan LocationService termasuk dalam @Service karena bertugas mengolah data dan menjalankan logika bisnis sebelum data dikirim ke atau diambil dari database.
@Controller (Endpoint API) Class seperti UserController, DonorController, DonationController, BloodRequestController, ServiceHistoryController, dan LocationController termasuk dalam @Controller karena berfungsi menerima dan merespons request dari pengguna melalui API menggunakan metode HTTP seperti GET, POST, PUT, dan DELETE.
3.5 Rancangan REST API 
Rancangan REST API adalah proses merancang antarmuka komunikasi antara client (seperti web atau mobile) dengan server menggunakan prinsip REST (Representational State Transfer). Dalam rancangan ini, data dan fungsi sistem diakses melalui endpoint berbasis URL, serta menggunakan metode HTTP seperti GET, POST, PUT, dan DELETE untuk melakukan operasi terhadap resource (misalnya data user, donor, atau permintaan). Tujuan dari perancangan REST API adalah agar sistem dapat berkomunikasi secara terstruktur, mudah dipahami, fleksibel, dan dapat digunakan oleh berbagai platform secara konsisten.

Users
Pada entitas User, endpoint digunakan untuk mengelola data akun yang berperan sebagai pengguna sistem. Melalui GET /api/users, sistem dapat menampilkan seluruh data user, sedangkan GET /api/users/{id} digunakan untuk melihat detail user tertentu. Proses penambahan user baru dilakukan melalui POST /api/users dengan mengirimkan data seperti nama, email, dan password. Jika terdapat perubahan data, maka dapat dilakukan menggunakan PUT /api/users/{id}, dan apabila data sudah tidak diperlukan lagi dapat dihapus melalui DELETE /api/users/{id}. Entitas ini menjadi dasar karena berhubungan dengan peran pengguna dalam sistem. 
Donor
Pada entitas Donor, endpoint digunakan untuk mengelola data pendonor darah yang terdaftar dalam sistem. Endpoint GET /api/donor digunakan untuk menampilkan seluruh data donor, sedangkan GET /api/donor/{id} digunakan untuk melihat detail donor berdasarkan ID tertentu. Penambahan data donor dilakukan melalui POST /api/donor dengan mengirimkan informasi seperti golongan darah, alamat, jenis kelamin, dan data pendukung lainnya. Jika terdapat perubahan data donor, maka dapat dilakukan melalui PUT /api/donor/{id}. Sementara itu, DELETE /api/donor/{id} digunakan untuk menghapus data donor yang sudah tidak digunakan.
Donation
Pada entitas Donation, endpoint digunakan untuk mencatat aktivitas pelayanan masyarakat yang dilakukan oleh pendonor. Endpoint GET /api/donation berfungsi untuk menampilkan seluruh data donasi, sedangkan GET /api/donation/{donor} digunakan untuk melihat riwayat donor berdasarkan ID pendonor. Proses pencatatan donor dilakukan melalui POST /api/donation, di mana sistem akan menyimpan data donasi sekaligus memperbarui data layanan sesuai dengan jumlah yang didonorkan. Entitas ini berperan penting dalam menjaga ketersediaan data layanan dalam sistem. 
Lokasi
Pada entitas Lokasi, endpoint digunakan untuk mengelola data tempat pelaksanaan pelayanan masyarakat. Endpoint GET /api/lokasi digunakan untuk menampilkan seluruh lokasi yang tersedia, sedangkan GET /api/lokasi/{id} digunakan untuk melihat detail lokasi tertentu. Penambahan lokasi baru dilakukan melalui POST /api/lokasi, sementara pembaruan data lokasi seperti alamat atau koordinat dilakukan melalui PUT /api/lokasi/{id}. Jika lokasi sudah tidak digunakan, maka dapat dihapus melalui DELETE /api/lokasi/{id}. 
Request Darah
Pada entitas Request Darah, endpoint digunakan untuk menangani permohonan layanan dari pengguna atau instansi. Endpoint GET /api/requestdarah berfungsi untuk mengambil seluruh data permintaan, sedangkan GET /api/requestdarah/{id} digunakan untuk melihat detail permintaan tertentu. Untuk membuat permintaan baru digunakan POST /api/requestdarah, di mana sistem akan melakukan pengecekan data layanan dan menentukan status awal apakah dapat diproses atau masih pending. Selanjutnya, PUT /api/requestdarah/{id} digunakan untuk memperbarui status permintaan sesuai perkembangan proses, dan DELETE /api/requestdarah/{id} digunakan untuk menghapus data permintaan jika diperlukan. 
BAB IV
IMPLEMENTASI SISTEM
4.1 Implementasi Entity
Entity merupakan class yang digunakan untuk merepresentasikan tabel pada database. Setiap entity dibuat menggunakan anotasi @Entity dari Spring Data JPA sehingga dapat terhubung langsung dengan tabel pada database MySQL.
Pada Sistem Pelayanan Donor Darah Masyarakat terdapat beberapa entity utama yaitu User, Donor, Location, BloodRequest, BloodStock, Donation, Complaint, dan ServiceHistory. Masing-masing entity memiliki atribut yang sesuai dengan struktur tabel pada database serta relasi antar entity menggunakan anotasi seperti @OneToOne, @OneToMany, dan @ManyToOne.
Implementasi entity bertujuan untuk mempermudah proses penyimpanan, pengambilan, dan pengelolaan data pada sistem.
4.2 Implementasi Repository
Repository digunakan sebagai lapisan akses data yang berfungsi menghubungkan aplikasi dengan database. Pada sistem ini digunakan Spring Data JPA sehingga proses pengolahan data dapat dilakukan tanpa menuliskan query SQL secara manual.
Beberapa repository yang digunakan antara lain UserRepository, DonorRepository, LocationRepository, BloodRequestRepository, BloodStockRepository, DonationRepository, ComplaintRepository, dan ServiceHistoryRepository.
Melalui repository, sistem dapat melakukan operasi Create, Read, Update, dan Delete (CRUD) terhadap data yang tersimpan pada database.
4.3 Implementasi Service
Service merupakan lapisan yang berfungsi menjalankan logika bisnis sistem. Service bertindak sebagai penghubung antara Controller dan Repository.
Pada sistem ini terdapat beberapa service seperti UserService, DonorService, DonationService, BloodRequestService, BloodStockService, ComplaintService, ServiceHistoryService, dan LocationService.
Proses seperti validasi data, pengolahan data donor, pembaruan stok darah, serta pengelolaan status pelayanan dilakukan pada layer service sebelum data disimpan ke database.
4.4 Implementasi Controller
Controller berfungsi menerima request dari client dan mengirimkan response kembali kepada pengguna melalui REST API.
Pada sistem ini digunakan beberapa controller seperti UserController, DonorController, DonationController, BloodRequestController, BloodStockController, ComplaintController, ServiceHistoryController, dan LocationController.
Setiap controller menggunakan anotasi @RestController dan @RequestMapping untuk mengatur endpoint yang tersedia pada sistem.
4.5 Implementasi REST API
REST API digunakan sebagai media komunikasi antara client dan server menggunakan protokol HTTP.
Metode yang digunakan pada sistem meliputi:
GET digunakan untuk mengambil data.
POST digunakan untuk menambahkan data.
PUT digunakan untuk memperbarui data.
DELETE digunakan untuk menghapus data.

4.6 Implementasi Layered Architecture
Sistem Pelayanan Donor Darah Masyarakat menerapkan konsep Layered Architecture yang terdiri dari beberapa lapisan yaitu Controller Layer, Service Layer, Repository Layer, dan Database Layer.
Controller Layer bertugas menerima request dari pengguna. Service Layer bertugas menjalankan logika bisnis aplikasi. Repository Layer berfungsi mengakses database menggunakan Spring Data JPA. Sedangkan Database Layer digunakan untuk menyimpan seluruh data sistem.
Dengan penerapan Layered Architecture, kode program menjadi lebih terstruktur, mudah dikembangkan, serta mempermudah proses pemeliharaan aplikasi.
4.7 Implementasi Exception Handling
Exception Handling digunakan untuk menangani kesalahan yang terjadi selama proses eksekusi aplikasi sehingga sistem tetap dapat berjalan dengan baik.
Pada sistem ini digunakan mekanisme try-catch dan custom exception untuk menangani berbagai kondisi kesalahan seperti data tidak ditemukan, input tidak valid, serta kesalahan saat proses penyimpanan data ke database.
Dengan adanya Exception Handling, sistem dapat memberikan pesan kesalahan yang lebih jelas kepada pengguna dan mencegah terjadinya crash pada aplikasi.
4.8 Hasil Implementasi Sistem
Setelah seluruh komponen sistem berhasil diimplementasikan, aplikasi dapat menjalankan seluruh fitur utama yang telah dirancang. Fitur tersebut meliputi autentikasi pengguna, manajemen data donor, pengelolaan lokasi donor darah, manajemen stok darah, permintaan darah, pengaduan masyarakat, riwayat pelayanan, serta pencatatan aktivitas donor darah.
Implementasi yang dilakukan menggunakan Spring Boot, Spring Data JPA, REST API, dan MySQL berhasil mendukung kebutuhan sistem secara keseluruhan. Seluruh modul dapat saling berinteraksi dengan baik dan mampu menjalankan proses bisnis sesuai dengan perancangan yang telah dibuat.

Pengujian login dan registrasi dilakukan untuk memastikan pengguna dapat membuat akun baru serta masuk ke dalam sistem menggunakan akun yang telah terdaftar. 6.3 Pengujian Data Donor
Pengujian data donor dilakukan untuk memastikan fitur pengelolaan data donor dapat berjalan dengan baik. Pengujian meliputi proses penambahan data donor baru, menampilkan data donor, memperbarui data donor, serta menghapus data donor. Hasil pengujian menunjukkan bahwa seluruh proses tersebut dapat dijalankan tanpa kendala dan data tersimpan dengan baik pada database.
6.4 Pengujian Lokasi Donor Darah
Pengujian lokasi donor darah dilakukan untuk memastikan sistem mampu mengelola informasi lokasi donor darah yang tersedia. Pengujian mencakup proses penambahan lokasi baru, menampilkan daftar lokasi, memperbarui informasi lokasi, serta menghapus data lokasi. Berdasarkan hasil pengujian, seluruh fungsi berjalan dengan baik sesuai dengan kebutuhan sistem.
6.5 Pengujian Stok Darah
Pengujian stok darah dilakukan untuk memastikan data ketersediaan darah dapat dikelola dengan baik oleh sistem. Proses pengujian meliputi penambahan stok darah, menampilkan data stok darah, serta memperbarui jumlah stok darah berdasarkan golongan darah tertentu. Hasil pengujian menunjukkan bahwa data stok darah dapat diproses dan ditampilkan dengan benar.
6.6 Pengujian Permintaan Darah
Pengujian permintaan darah dilakukan untuk memastikan pengguna dapat mengajukan permintaan darah melalui sistem. Selain itu, pengujian juga dilakukan terhadap proses perubahan status permintaan darah dari pending menjadi diproses hingga selesai. Hasil pengujian menunjukkan bahwa sistem mampu mengelola permintaan darah dengan baik dan status pelayanan dapat diperbarui sesuai kondisi yang terjadi.
6.7 Pengujian Pengaduan Masyarakat
Pengujian pengaduan masyarakat dilakukan untuk memastikan pengguna dapat mengirimkan keluhan, kritik, maupun saran melalui sistem. Data pengaduan yang dikirim berhasil disimpan ke dalam database dan dapat ditampilkan kembali oleh administrator untuk ditindaklanjuti. Seluruh proses pengelolaan pengaduan berjalan dengan baik sesuai kebutuhan sistem.
6.8 Hasil Pengujian Sistem
Berdasarkan seluruh pengujian yang telah dilakukan, Sistem Pelayanan Donor Darah Masyarakat berhasil menjalankan seluruh fitur utama sesuai dengan perancangan yang telah dibuat. Fitur autentikasi pengguna, pengelolaan donor, lokasi donor darah, stok darah, permintaan darah, pengaduan masyarakat, dan riwayat pelayanan dapat berfungsi dengan baik. Selain itu, seluruh endpoint REST API mampu memberikan respons yang sesuai dan data dapat tersimpan dengan baik pada database MySQL. Dengan demikian, sistem dinyatakan berhasil memenuhi kebutuhan fungsional yang telah ditetapkan.
BAB VII
KESIMPULAN DAN SARAN
7.1 Kesimpulan
Berdasarkan hasil perancangan, implementasi, deployment, dan pengujian yang telah dilakukan, dapat disimpulkan bahwa Sistem Pelayanan Donor Darah Masyarakat berhasil dibangun menggunakan bahasa pemrograman Java dengan framework Spring Boot serta database MySQL. Sistem ini mampu membantu masyarakat dalam memperoleh informasi dan layanan donor darah secara lebih cepat, mudah, dan terstruktur.
Penerapan konsep Object-Oriented Programming (OOP) seperti Encapsulation, Inheritance, Polymorphism, Interface, Abstract Class, dan Exception Handling berhasil diterapkan pada pengembangan sistem. Selain itu, penggunaan Layered Architecture membuat struktur aplikasi menjadi lebih terorganisir dan mudah untuk dikembangkan maupun dipelihara.
Sistem yang dibangun menyediakan berbagai fitur seperti autentikasi pengguna, manajemen data donor, informasi lokasi donor darah, pengelolaan stok darah, layanan permintaan darah, riwayat pelayanan, pengaduan masyarakat, dan riwayat donor. Seluruh fitur tersebut dapat berjalan dengan baik berdasarkan hasil pengujian yang telah dilakukan.
Proses deployment menggunakan Spring Boot dan Cloudflare Tunnel juga berhasil dilakukan sehingga aplikasi dapat diakses melalui jaringan internet menggunakan URL publik. Dengan demikian, sistem yang dikembangkan telah memenuhi kebutuhan fungsional yang ditetapkan serta mampu mendukung peningkatan pelayanan masyarakat dalam bidang donor darah.
7.2 Saran
Pengembangan sistem di masa mendatang dapat dilakukan dengan menambahkan fitur notifikasi otomatis untuk memberikan informasi kepada pengguna terkait status permintaan darah maupun jadwal donor darah. Selain itu, sistem juga dapat dikembangkan dengan menambahkan fitur chatbot, integrasi peta yang lebih interaktif, serta sistem keamanan yang lebih baik seperti JWT Authentication.
Pengembangan aplikasi mobile berbasis Android atau iOS juga dapat menjadi alternatif untuk meningkatkan kemudahan akses bagi masyarakat. Dengan adanya pengembangan lebih lanjut, diharapkan sistem dapat memberikan pelayanan donor darah yang lebih efektif, efisien, dan bermanfaat bagi masyarakat luas.
