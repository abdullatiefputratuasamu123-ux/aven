BAB I
PENDAHULUAN
1.1 Latar Belakang
Pelayanan publik merupakan salah satu tanggung jawab utama pemerintah
kepada masyarakat. Namun, hingga saat ini banyak instansi pemerintah di tingkat
kelurahan dan kecamatan yang masih mengandalkan proses manual dalam mengelola
pengaduan warga maupun permohonan layanan administrasi seperti pembuatan surat
keterangan, surat domisili, dan izin usaha mikro.
Proses manual tersebut menimbulkan sejumlah permasalahan, di antaranya:
antrean panjang di kantor pelayanan, risiko kehilangan berkas, ketidakjelasan status
proses, serta rendahnya transparansi bagi warga selaku pemohon. Kondisi ini
berdampak pada menurunnya kepercayaan masyarakat terhadap kualitas pelayanan
pemerintah setempat.
Perkembangan teknologi informasi, khususnya platform berbasis web dan
REST API, membuka peluang untuk menghadirkan solusi digital yang mampu
memusatkan, menvirtualkan, dan meningkatkan efisiensi proses pelayanan tersebut.
Dengan memanfaatkan paradigma Object Oriented Programming (OOP) melalui
framework Spring Boot, dimungkinkan untuk membangun sistem yang terstruktur,
skalabel, dan mudah dipelihara.
Berdasarkan latar belakang di atas, kami mengusulkan pengembangan sistem
bernama "SmartPelayanan (SPLM)" — sebuah platform backend satu pintu digital
yang memungkinkan warga mengajukan pengaduan dan permohonan layanan
administrasi secara online, sementara petugas/admin dapat memantau, memproses,
dan memperbarui status secara real-time melalui antarmuka REST API.
1.2 Rumusan Masalah
Berdasarkan uraian latar belakang di atas, rumusan masalah yang dihadapi adalah
sebagai berikut:
1. Bagaimana membangun platform backend (REST API) yang dapat menjadi
satu pintu terpadu bagi warga untuk menyampaikan pengaduan dan
mengajukan permohonan layanan administrasi secara digital?
2. Bagaimana menyediakan mekanisme pelacakan status (tracking) permohonan
secara real-time sehingga warga dapat mengetahui perkembangan
pengajuannya tanpa harus mendatangi kantor?
Halaman 4 dari 18
3. Bagaimana mengelola dan mengarsipkan data pengaduan beserta dokumen
pendukung secara sistematis agar admin dapat memonitor dan merespons
dengan cepat?
4. Bagaimana menerapkan prinsip-prinsip OOP (Encapsulation, Inheritance,
Polymorphism, Abstraction) secara nyata dalam arsitektur sistem
SmartPelayanan?
1.3 Tujuan Pengembangan
Tujuan dari pengembangan sistem SmartPelayanan adalah:
1. Membangun aplikasi backend REST API berbasis Spring Boot yang
terstruktur menggunakan pola Layered Architecture (Controller – Service –
Repository).
2. Mengimplementasikan manajemen pengaduan warga yang mencakup operasi
CRUD lengkap beserta pembaruan status proses.
3. Mengimplementasikan modul layanan administrasi yang mendukung
pengajuan dan pengelolaan dokumen permohonan warga.
4. Menerapkan konsep OOP secara terstruktur dan terdokumentasi sesuai
standar penilaian UTS Praktikum OOP.
5. Menyediakan dokumentasi perancangan sistem yang komprehensif meliputi
ERD, Class Diagram, dan spesifikasi API.
1.4 Batasan Sistem
Agar pengembangan tetap terfokus, sistem SmartPelayanan dibatasi pada hal-hal
berikut:
• Lingkup pengembangan mencakup layer Backend (REST API) saja;
antarmuka pengguna (frontend) tidak termasuk dalam cakupan proyek ini.
• Sistem tidak mengintegrasikan modul pembayaran (payment gateway)
apapun.
• Autentikasi dan otorisasi diimplementasikan menggunakan mekanisme
Role-Based Access Control (RBAC) sederhana dengan role: WARGA dan
ADMIN.
• Penyimpanan dokumen pendukung menggunakan referensi path/URL file;
belum mencakup penyimpanan cloud (object storage).
• Sistem diuji secara lokal menggunakan Postman/Insomnia; belum mencakup
pengujian beban (load testing).

• Bagi Warga: Kemudahan mengajukan pengaduan atau permohonan layanan

BAB II
LANDASAN TEORI
2.1 Konsep Object Oriented Programming (OOP)
Object Oriented Programming (OOP) adalah paradigma pemrograman yang
mengorganisasikan kode ke dalam unit-unit yang disebut class dan object. Paradigma
ini berfokus pada pemodelan entitas dunia nyata ke dalam bentuk objek yang
memiliki atribut (data) dan perilaku (method). Empat pilar utama OOP yang
diterapkan dalam SmartPelayanan adalah:
a. Encapsulation (Enkapsulasi)
Encapsulation adalah mekanisme penyembunyian detail implementasi internal
sebuah class dari dunia luar. Dalam SmartPelayanan, seluruh field pada class Entity
seperti User dan Pengaduan dideklarasikan sebagai private, dan akses dari luar
dilakukan melalui method getter dan setter. Hal ini mencegah manipulasi data secara
tidak terkontrol dan menjaga integritas data objek.
b. Inheritance (Pewarisan)
Inheritance memungkinkan sebuah class (subclass) mewarisi atribut dan
method dari class lain (superclass). Dalam sistem ini, dirancang sebuah abstract class
bernama BaseEntity yang memiliki field audit umum berupa id (UUID), createdAt,
dan updatedAt. Class-class entitas utama seperti User, Pengaduan, dan
LayananAdministrasi meng-extend BaseEntity sehingga otomatis memiliki field audit
tersebut tanpa perlu mendefinisikan ulang.
c. Polymorphism (Polimorfisme)
Polymorphism memungkinkan satu method berperilaku berbeda berdasarkan
konteks objek yang memanggilnya. Dalam SmartPelayanan, polimorfisme diterapkan
melalui mekanisme interface: setiap modul bisnis mendefinisikan kontrak atau
interface (misal: PengaduanService), dan implementasi nyatanya ditempatkan di kelas
terpisah (misal: PengaduanServiceImpl). Spring Boot secara dinamis menginjeksikan
implementasi tersebut (Dependency Injection), sehingga perubahan implementasi
tidak mempengaruhi layer Controller.
d. Abstraction (Abstraksi)
Abstraction adalah proses menyembunyikan detail kompleks dan hanya
mengekspos fungsionalitas esensial kepada pengguna. Dalam sistem ini, abstraksi
diwujudkan melalui penggunaan interface sebagai service contract. Layer Controller
hanya berinteraksi dengan interface service tanpa mengetahui detail implementasinya.

Selain itu, penggunaan abstract class BaseEntity juga merupakan contoh abstraksi
untuk menghindari duplikasi kode audit.
2.2 Spring Boot dan Arsitektur Berlapis
Spring Boot adalah framework berbasis Java yang menyederhanakan
pengembangan aplikasi berbasis Spring Framework dengan konfigurasi otomatis
(auto-configuration) dan prinsip Convention over Configuration. Spring Boot
memungkinkan pengembang untuk langsung fokus pada logika bisnis tanpa harus
mengonfigurasi server atau dependency secara manual.
SmartPelayanan mengadopsi Layered Architecture (Arsitektur Berlapis) tiga lapis:
• Controller Layer (@RestController): Menerima HTTP request, memvalidasi
input, mendelegasikan proses ke Service Layer, lalu mengemas response
dalam format JSON.
• Service Layer (@Service + Interface): Memuat seluruh logika bisnis
aplikasi. Setiap service direpresentasikan sebagai interface yang
diimplementasikan oleh kelas ServiceImpl.
• Repository Layer (@Repository): Berinteraksi langsung dengan database
menggunakan Spring Data JPA. Meng-extend JpaRepository<Entity, ID>
untuk mendapatkan operasi CRUD built-in.
2.3 REST API dan HTTP
REST (Representational State Transfer) adalah gaya arsitektur komunikasi
antara client dan server melalui protokol HTTP. API RESTful menggunakan HTTP
method yang bermakna dan mengidentifikasi sumber daya (resource) melalui URL.
SmartPelayanan mengimplementasikan HTTP method berikut:
• GET — Mengambil/membaca data resource.
• POST — Membuat resource baru.
• PUT — Memperbarui data resource secara menyeluruh.
• PATCH — Memperbarui sebagian field resource (misal: status saja).
• DELETE — Menghapus resource.
Seluruh response dari API SmartPelayanan menggunakan format JSON
(JavaScript Object Notation) dengan struktur baku yang terdiri dari field: status,
message, dan data.

BAB III
PERANCANGAN SISTEM
3.1 Gambaran Umum Sistem
SmartPelayanan adalah sistem informasi backend yang berfungsi sebagai
jembatan digital antara warga dan petugas administrasi. Sistem ini terdiri dari dua
aktor utama: WARGA (pengguna yang mengajukan pengaduan/layanan) dan
ADMIN (petugas yang memproses dan merespons setiap permohonan). Berikut
adalah gambaran alur penggunaan sistem secara umum:
• Warga melakukan registrasi dan login untuk mendapatkan akses ke sistem.
• Warga dapat mengajukan pengaduan baru atau permohonan layanan
administrasi.
• Warga dapat memantau status pengaduan/permohonannya secara real-time.
• Admin menerima notifikasi pengajuan baru dan dapat memperbarui status
pengaduan.
• Admin dapat mengelola data kategori layanan dan merekap seluruh data
pengaduan.

3.5 Alur Kerja Sistem (Workflow)
Berikut adalah alur kerja utama penggunaan SmartPelayanan dari perspektif kedua
aktor:
Alur Pengajuan Pengaduan oleh Warga:
1. Warga melakukan registrasi melalui POST /api/v1/auth/register.
2. Warga login melalui POST /api/v1/auth/login dan mendapatkan JWT token.
3. Warga mengajukan pengaduan melalui POST /api/v1/pengaduan dengan
melampirkan detail masalah.
4. Sistem otomatis menetapkan status = BARU dan menyimpan data ke
database.
5. Warga dapat memantau status pengaduan melalui GET
/api/v1/pengaduan/{id}.
Alur Penanganan Pengaduan oleh Admin:
1. Admin login menggunakan akun admin dan mendapatkan JWT token.
2. Admin melihat seluruh daftar pengaduan melalui GET /api/v1/pengaduan.
3. Admin memproses pengaduan dan mengubah status menjadi DIPROSES
melalui PATCH /api/v1/pengaduan/{id}/status.
4. Admin menambahkan respon/tanggapan melalui POST
/api/v1/pengaduan/{id}/respon.
5. Setelah selesai ditangani, admin mengubah status menjadi SELESAI.
Seluruh perubahan status tercatat di tb_riwayat_status.

PENUTUP
4.1 Kesimpulan
SmartPelayanan (SPLM) adalah solusi backend digital yang dirancang untuk
memodernisasi proses pengaduan dan pelayanan administrasi masyarakat. Melalui
proposal ini, telah dijabarkan perancangan sistem yang komprehensif mencakup:
• Latar belakang dan justifikasi kebutuhan sistem berbasis masalah pelayanan
manual yang ada.
• Landasan teori OOP (Encapsulation, Inheritance, Polymorphism,
Abstraction) beserta penerapannya dalam Spring Boot.
• Desain ERD dengan 6 tabel yang saling berelasi secara terstruktur dan
ternormalisasi.
• Rancangan Class Diagram dengan 10 class yang menerapkan pola Layered
Architecture dan prinsip OOP secara konsisten.
• Spesifikasi REST API yang mencakup 23 endpoint terkelompok dalam 5
modul fungsional.
Dengan rancangan yang terstruktur ini, implementasi sistem diharapkan dapat
berjalan lancar dan memenuhi seluruh kriteria penilaian Ujian Tengah Semester
Praktikum OOP.
